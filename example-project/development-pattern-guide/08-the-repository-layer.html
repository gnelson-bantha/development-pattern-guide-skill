Chapter 8

The Repository Layer

A narrow interface over persistence: the contract it exposes, the ORM it hides, and the substitution that hiding buys.

The Contract

An Interface, Then an Implementation

Defining the boundary before the detail.

The repository is the seam between the application and the store. It is defined as an interface first, because the interface is what callers depend on, and the implementation is a detail they never see. The interface lists only the operations the application actually performs.

public interface IRepository
{
    void AddRange(IEnumerable<Entity> entities);
    List<Entity> GetAll();
    int Count();
}
public class Repository(DataContext context) : IRepository
{
    public void AddRange(IEnumerable<Entity> entities)
    {
        context.Entities.AddRange(entities);
        context.SaveChanges();
    }

    public List<Entity> GetAll() => [.. context.Entities];

    public int Count() => context.Entities.Count();
}
Why It Exists

Hiding the ORM Behind a Small Surface

The reason a repository sits between the service and the context.

The Orchestration Service could call the Database Context directly. The repository exists so it does not. Without the repository, the service would reference entity sets, query operators, and the save call, and it would be coupled to the ORM. Every layer that persisted anything would carry that coupling, and a test of the service could not run without the ORM.

The repository collapses all of that into three methods. The service depends on IRepository and knows nothing about how persistence happens. The ORM is named in exactly one class. The surface the rest of the program sees is small enough to re-implement by hand, which is precisely what a test does.

The Payoff

Swapping the Store Without Touching Callers

What the narrow interface makes possible, and what it costs.

Because callers depend on the interface, any class that satisfies IRepository is a valid substitute. A test replaces it with an in-memory list and never starts a database:

public class InMemoryRepository : IRepository
{
    private readonly List<Entity> _store = [];

    public void AddRange(IEnumerable<Entity> entities) => _store.AddRange(entities);
    public List<Entity> GetAll() => [.. _store];
    public int Count() => _store.Count;
}

The same property lets production move from the in-memory provider to a real database by changing the context registration, with the repository and everything above it untouched. The cost of the layer is that it is one more interface and one more class, and that the repository can drift toward exposing ORM concepts (returning queryables, leaking change tracking) if you let it. Keep the interface expressed in domain terms (add these, get all, count) and the seam stays clean.

Save Inside the Method

AddRange calls SaveChanges itself, so a caller gets a complete unit of work and never has to remember a separate commit step.

Return Materialized Lists

GetAll returns a concrete list, not a deferred query, so no ORM query object escapes the repository to be evaluated elsewhere.

Domain-Term Methods

Method names describe what the application wants, not how the store does it, which keeps callers ignorant of the backing technology.
