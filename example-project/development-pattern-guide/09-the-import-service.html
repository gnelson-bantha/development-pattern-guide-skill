Chapter 9

The Import Service

The boundary to the outside world: reading an external source, deserializing it into entities, and failing clearly when the source is missing.

The Contract

One Method, One Job

The interface that the orchestration layer depends on.

The import service has a single responsibility: turn an external source into a list of domain models. Its interface says exactly that and nothing more, so callers cannot become entangled in how the reading happens.

public interface IImportService
{
    List<Entity> LoadFromSource();
}
using System.Text.Json;

public class ImportService : IImportService
{
    private static readonly JsonSerializerOptions SerializerOptions = new()
    {
        PropertyNameCaseInsensitive = true
    };

    private readonly string _sourcePath =
        Path.Combine(AppContext.BaseDirectory, "Data", "entities.json");

    public List<Entity> LoadFromSource()
    {
        if (!File.Exists(_sourcePath))
        {
            throw new FileNotFoundException(
                $"Could not find the data file at '{_sourcePath}'.",
                _sourcePath);
        }

        string json = File.ReadAllText(_sourcePath);
        List<Entity>? entities =
            JsonSerializer.Deserialize<List<Entity>>(json, SerializerOptions);

        return entities ?? [];
    }
}
Boundary

Containing the Outside World

Why all the messy I/O lives in this one class.

Everything uncertain about the input is contained here: the file may not exist, the path is environment-dependent, the JSON casing may not match the C# property names, and deserialization can return null. Each of these is handled in this class so that no other layer has to think about them. The Orchestration Service receives a clean List<Entity> and never learns that the data came from a file.

This containment is what makes the source replaceable. To read from an HTTP endpoint instead of a file, you write a new class implementing IImportService that fetches and deserializes a response. The orchestration layer does not change, because it depends on the interface. The boundary class absorbs the change, which is the entire reason the boundary is a separate layer.

Failure Handling

Failing at the Edge, Clearly

Validating input where it enters and normalizing the empty case.

The service checks for the file before reading and throws a specific exception naming the path it looked for. This is deliberate placement: input is validated at the boundary where it enters the system, so a missing source produces a clear error here rather than a confusing null reference three layers later. Validating at the edge is a general rule of the pattern, and the import service is the edge for input.

The final line normalizes the one ambiguous result the serializer can return. Deserializing a literal null yields a null list, which is not the same as an empty source. Coercing it to an empty list means every caller can assume a non-null collection, removing a null check from every layer above. Handling the awkward case once, at the boundary, keeps the upper layers straight.

Base Directory Path

The source path is built from the application's base directory, so the file is found regardless of the working directory the program was launched from.

Case-Insensitive Binding

The serializer options accept fields whose casing differs from the C# properties, so the external format and the model naming can evolve independently.

Null Coalesced to Empty

A null deserialization result becomes an empty list, giving every caller a guaranteed non-null collection to iterate.
