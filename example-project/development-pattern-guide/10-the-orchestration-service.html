Chapter 10

The Orchestration Service

The use case that composes the lower layers: receiving its collaborators by injection and calling them in order.

The Contract

The Service and Its Injected Collaborators

Two dependencies, both interfaces, both arriving through the constructor.

The orchestration service is where the use case lives. It does not read files and it does not talk to the ORM; it coordinates the layers that do. It receives both of those layers as interfaces through its constructor and never constructs either one.

public interface IOrchestrationService
{
    void Run();
}
public class OrchestrationService(
    IImportService importService,
    IRepository repository) : IOrchestrationService
{
    public void Run()
    {
        List<Entity> imported = importService.LoadFromSource();
        repository.AddRange(imported);

        List<Entity> stored = repository.GetAll();

        Console.WriteLine($"Loaded {repository.Count()} entities from the store:");
        Console.WriteLine();

        foreach (Entity entity in stored)
        {
            Console.WriteLine($"Id:       {entity.Id}");
            Console.WriteLine($"Name:     {entity.Name}");
            Console.WriteLine($"Category: {entity.Category}");
            Console.WriteLine($"Value:    {entity.Value:N0}");
            Console.WriteLine(new string('-', 40));
        }
    }
}
Constructor Injection

Depending on Interfaces, Not Classes

The single rule that keeps this layer testable.

The constructor parameters are IImportService and IRepository, not ImportService and Repository. This is the heart of the pattern applied to the top layer. The service is written entirely against abstractions, so the concrete classes that satisfy them can be chosen elsewhere. The service is unaware of files, JSON, the ORM, or the in-memory provider; it knows only the two contracts it was handed.

The immediate benefit is a test that needs no infrastructure. Construct the service with a fake importer that returns a fixed list and the in-memory repository from Chapter 8, call Run, and assert on the result. No file, no database. The service cannot tell the fakes from the real implementations, because it only ever saw interfaces. That is what "depend on abstractions" buys, and it is why this layer holds the logic worth testing.

Control Flow

Owning the Sequence, Not the Detail

Where the order of operations lives and where it does not.

The Run method is the one place that knows the order of the use case: import, then persist, then read back, then format. That sequence is real logic, and concentrating it here is deliberate. Each step delegates to a layer that owns the how, while this method owns the what and the when. The service stays readable because it reads like a description of the use case rather than a description of file or database mechanics.

Notice what the method does not contain. There is no path handling, no serializer configuration, no save call, no query. Each of those belongs to a lower layer and would, if pulled up here, fuse the orchestration back together with the detail and undo the separation. The discipline is to keep this method a sequence of intentions and push every mechanism down a layer.

Reads Back After Writing

The service retrieves entities from the repository rather than reusing the imported list, so the output reflects what the store actually holds.

Formatting Lives Here

Turning entities into output lines is part of the use case, so it sits in the service rather than on the model or in the repository.

No Infrastructure Knowledge

The method names a sequence of operations on two interfaces and references no file, database, or serializer concept.
