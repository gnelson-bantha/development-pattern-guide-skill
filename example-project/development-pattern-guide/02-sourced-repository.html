Chapter 2

Sourced Repository

The pattern came from a real app called WorldLocations. This is the only chapter that uses its real names, so you can see the concrete thing before we abstract it.

Real names ahead

Every other chapter uses generalized names like Import Service and Repository. Here we use the actual identifiers from the source project, so you can map the lesson back to working code.

The Project

WorldLocations at a Glance

A console app that loads a handful of cities, stashes them in a database, and prints them back out.

WorldLocations is a .NET 10 console application. Run it and it reads a bundled Data/locations.json file full of cities, populations, and countries, loads those rows into an Entity Framework Core database, reads them back, and writes them to the terminal. That's the entire user-facing behavior. The interesting part isn't what it does; it's how the few hundred lines are arranged.

The dependencies are deliberately small. The project file pulls in two packages and nothing else:

<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
    <RootNamespace>WorldLocations</RootNamespace>
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="Microsoft.EntityFrameworkCore.InMemory" Version="10.0.0" />
    <PackageReference Include="Microsoft.Extensions.DependencyInjection" Version="10.0.0" />
  </ItemGroup>

  <ItemGroup>
    <None Update="Data/locations.json">
      <CopyToOutputDirectory>PreserveNewest</CopyToOutputDirectory>
    </None>
  </ItemGroup>

</Project>

Two packages tell you almost everything. Microsoft.EntityFrameworkCore.InMemory is the database. Microsoft.Extensions.DependencyInjection is the wiring. The None Update block copies the JSON file next to the compiled binaries so the running program can find it. Nullable reference types and implicit usings are on, which keeps the source terse and the null-handling honest.

The Folders

How the Source Is Organized

Five folders, and each one is a layer. The directory tree is already the architecture diagram.

Open the project and the structure reads like a table of contents for this guide. Each folder is a responsibility:

  • Models/ holds Location, the data class.
  • Data/ holds LocationDbContext and the seed file locations.json.
  • Repositories/ holds ILocationRepository and LocationRepository.
  • Services/ holds two pairs: ILocationImportService / LocationImportService and ILocationService / LocationService.
  • Program.cs at the root is the entry point and the composition root.

The pairing of interface plus implementation shows up everywhere. That repetition is the pattern leaving fingerprints: every layer is defined first as a contract and only then as code that satisfies it.

The Types

The Classes That Do the Work

Six small types, each with a single responsibility and a clear edge against its neighbors.

Location is the simplest: four properties (Id, CityName, CountryName, Population) decorated with both [JsonPropertyName] attributes for deserialization and a [Key] attribute for EF Core. One class serves two systems at once, which keeps the data shape in a single place.

LocationImportService owns input. It builds a path with Path.Combine(AppContext.BaseDirectory, "Data", "locations.json"), checks the file exists and throws a FileNotFoundException if it doesn't, then deserializes with case-insensitive JSON options. Its only public method is LoadFromJson(), and it returns an empty list rather than null if the file parses to nothing.

LocationDbContext is the EF Core context. It exposes DbSet<Location> Locations and configures the entity in OnModelCreating: the key, plus required constraints on CityName and CountryName. It takes its DbContextOptions through a primary constructor, which is what lets the container decide it should be the in-memory provider.

LocationRepository implements ILocationRepository and exposes exactly three operations: AddRange, GetAll, and Count. It calls SaveChanges right after adding, so callers never think about change tracking. Nothing above it imports EF Core.

LocationService implements ILocationService and is the conductor. Its ImportAndDisplay() method asks the import service to load, hands the result to the repository, reads it back, and prints each record with a divider line. It depends on the two interfaces and constructs nothing.

The contract every layer keeps

No layer constructs the layer it depends on. Each one accepts its collaborators through its constructor and talks to them only through interfaces, so the concrete choices all defer to one place.

The seam that makes it swappable

Because persistence sits behind ILocationRepository and the database behind DbContextOptions, the in-memory provider could become SQLite or SQL Server without LocationService noticing.

The Entry Point

What Program.cs Pulls Together

Twenty-odd lines register the parts, reset the database, and pull the trigger.

The whole program is started by top-level statements in Program.cs. It creates a ServiceCollection, registers LocationDbContext against the in-memory provider, registers the three services as scoped, and builds a provider. Then it deletes and recreates the database for a clean run, resolves ILocationService, and calls ImportAndDisplay(). That single resolve triggers the container to build the entire object graph behind it.

ServiceCollection services = new();

services.AddDbContext<LocationDbContext>(options =>
    options.UseInMemoryDatabase("WorldLocationsDb"));

services.AddScoped<ILocationRepository, LocationRepository>();
services.AddScoped<ILocationImportService, LocationImportService>();
services.AddScoped<ILocationService, LocationService>();

using ServiceProvider provider = services.BuildServiceProvider();

LocationDbContext dbContext = provider.GetRequiredService<LocationDbContext>();
dbContext.Database.EnsureDeleted();
dbContext.Database.EnsureCreated();

ILocationService locationService = provider.GetRequiredService<ILocationService>();
locationService.ImportAndDisplay();

Everything the rest of this guide teaches generalizes from exactly these pieces. Location becomes the Domain Model, LocationImportService becomes the Import Service, LocationDbContext becomes the Persistence Context, LocationRepository becomes the Repository, LocationService becomes the Orchestration Service, and Program.cs becomes the Composition Root. With the concrete version in mind, let's line up the tools you'll need to build your own.
