Chapter 8

Repository Layer

The context can talk to a database. This layer makes sure nobody else has to. It's a thin wall, and the thinness is the point.

Step 1

The Smallest Useful Interface

List only the operations the program actually performs, and watch how much the rest of the code stops needing to know.

The repository is a thin wall between the rest of your code and the database. Code on this side asks for records; it never has to know they're sitting in an Entity Framework context. And the wall is defined, as always, by an interface first:

using Pipeline.Models;

namespace Pipeline.Repositories;

public interface IRepository
{
    void AddRange(IEnumerable<DataRecord> records);

    List<DataRecord> GetAll();

    int Count();
}

Three methods. That's the entire surface the rest of the program sees of persistence. There's a real temptation here to build the grand generic repository with Find, Update, Delete, paging, filtering, the works. Resist it. This pipeline adds records and reads them back, so the interface has exactly the methods this pipeline needs and not one more. A smaller interface is easier to implement, easier to fake in a test, and honest about what the system actually does. You can always add a method the day you genuinely need it.

Step 2

The EF Core Implementation

All the database knowledge in the entire program gets to live in this one class, and that containment is the whole reward.

Here's the implementation. It takes the context through its constructor, just like everything else takes its dependencies:

using Pipeline.Data;
using Pipeline.Models;

namespace Pipeline.Repositories;

public class PipelineRepository(PipelineDbContext dbContext) : IRepository
{
    public void AddRange(IEnumerable<DataRecord> records)
    {
        dbContext.Records.AddRange(records);
        dbContext.SaveChanges();
    }

    public List<DataRecord> GetAll() => [.. dbContext.Records];

    public int Count() => dbContext.Records.Count();
}

This is the only class in the whole program that mentions DbContext, AddRange, or SaveChanges. That's the payoff of the repository pattern stated plainly: every line of database-specific code is trapped inside this one file. If you decide tomorrow to store records in a flat file or call a web API instead, you write a new class that implements IRepository and nothing above it notices. The seam holds.

The [.. dbContext.Records] in GetAll is a collection expression that materializes the DbSet into a real List. That detail matters more than it looks. A DbSet is lazy; enumerate it and it hits the database. By spreading it into a list right here, we run the query inside the repository and hand back plain in-memory records. Callers get a list they can iterate twice, filter, or count without accidentally firing more database queries. The laziness stays contained, exactly like everything else about the database.

Step 3

Owning the Save

Notice who calls SaveChanges, and notice who doesn't have to think about it ever again.

EF Core tracks changes in memory and only writes them when you call SaveChanges. That's flexible, but it's also a footgun: forget the call and your data silently never persists. So AddRange does the add and the save together, as one unit. A caller asks the repository to add records and trusts that when the method returns, they're stored. Full stop.

That's a deliberate decision about who carries the burden. The orchestration layer above shouldn't know that "adding" and "saving" are two separate steps in EF Core; that's a persistence detail, and persistence details belong down here. Bundling them keeps the contract simple ("add means stored") at the cost of a little flexibility (you can't batch many adds into one save). For this pipeline, simple wins, and that's usually the right instinct until a real performance need says otherwise.

With persistence sealed behind three methods, we finally have every worker the pipeline needs: a way to load records and a way to store and fetch them. What's missing is someone to run them in order. That's the Orchestration Service, next.
