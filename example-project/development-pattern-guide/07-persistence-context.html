Chapter 7

Persistence Context

We have records and a way to load them. Now they need somewhere to live. This is the layer that actually speaks database.

Step 1

The ORM Context

One class stands between your plain records and the database engine, and it's smaller than you'd expect.

Entity Framework Core is an object-relational mapper, which is a fancy way of saying it translates between C# objects and database rows so you don't write SQL by hand. The center of EF Core is the context, and ours is short:

using Microsoft.EntityFrameworkCore;
using Pipeline.Models;

namespace Pipeline.Data;

public class PipelineDbContext(DbContextOptions<PipelineDbContext> options)
    : DbContext(options)
{
    public DbSet<DataRecord> Records => Set<DataRecord>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<DataRecord>(entity =>
        {
            entity.HasKey(record => record.Id);
            entity.Property(record => record.Name).IsRequired();
            entity.Property(record => record.Category).IsRequired();
        });
    }
}

The DbSet<DataRecord> Records property is your handle on the table. Query it, add to it, count it, and EF Core turns those operations into database commands behind the scenes. Writing it as => Set<DataRecord>() instead of a settable property is a small modern touch that skips a nullability warning and makes the set read-only from the outside. The context is the engine; this DbSet is the steering wheel.

Step 2

Options Through the Constructor

Here's the move that makes the whole database swappable, and it hinges on the context not choosing its own provider.

Look closely at the constructor: PipelineDbContext(DbContextOptions<PipelineDbContext> options). The context accepts its configuration instead of creating it. It does not decide whether it's talking to an in-memory store, SQLite, or a full SQL Server. That decision arrives from outside, packaged up in the options object, and gets handed down to the base DbContext.

This is the same constructor-injection idea we keep meeting, applied to the database itself. The context says "tell me what provider to use" rather than "I'll use this provider." That inversion is what lets the composition root pick in-memory for a quick run and a real database for production without editing this file at all. The context stays honest about one thing, the shape of the data, and stays ignorant of another, where the data physically lives.

Why this matters for tests

Because the provider is injected, a test can spin up a throwaway in-memory database, run the real context against it, and verify behavior with no files and no cleanup. The production code and the test code share the exact same context class.

Step 3

Mapping Rules in OnModelCreating

The model already carries a key attribute, so why repeat the rules here? Because this is where the database-specific constraints belong.

OnModelCreating is where you tell EF Core how the Domain Model maps to a table. We declare the key explicitly with HasKey, and we mark Name and Category as required so the database refuses rows that leave them empty. You might wonder why we set the key here when the model already has a [Key] attribute. Both work, and you'd usually pick one. Keeping the persistence-specific rules in the context, though, has a real upside: the database's constraints live with the database concern, not scattered as attributes on a model that other layers also use.

The IsRequired calls are doing something the attributes weren't: enforcing a rule at the storage boundary. The Import Service might hand over a record with an empty name (maybe the source data was incomplete), and this constraint is the last line that says no. It's defense at the edge of persistence, separate from any validation the import side does or doesn't do.

The context knows how to store records, but we're not going to let the rest of the program touch it directly. That would leak EF Core everywhere. Instead, the next chapter wraps it in a much smaller interface: the Repository.
