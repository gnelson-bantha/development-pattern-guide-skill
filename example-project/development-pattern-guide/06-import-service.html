Chapter 6

Import Service

The records have a shape now. This layer is where messy outside data turns into those clean records, and where all the danger gets contained.

Step 1

An Interface First

Before the file-reading code, write the promise: one method that returns records and says nothing about where they came from.

We could jump straight to reading a file. Instead, write the contract first, because the contract is the part the rest of the system depends on, and it's deliberately silent about JSON:

using Pipeline.Models;

namespace Pipeline.Services;

public interface IImportService
{
    List<DataRecord> LoadFromSource();
}

Read that interface and you learn nothing about files, paths, or serialization. You learn that you can ask for records and you'll get a list back. That silence is the feature. The Orchestration Service will depend on IImportService, and because the interface never mentions JSON, the day you switch from a file to an HTTP call, the orchestrator's code doesn't change by a single character. The boundary was drawn at "give me records," not at "read this file," and that's the difference between a swappable layer and a brittle one.

Step 2

Locating the Source File

Finding the file sounds trivial until you remember the program runs from a different folder than the one you wrote it in.

Now the implementation. The first real problem is just finding the file, and it's sneakier than it looks. You can't hardcode a path from your machine, and you can't assume the current working directory, because a compiled program might be launched from anywhere. What you can rely on is where the binaries live, and the data file is copied right next to them at build time.

using System.Text.Json;
using Pipeline.Models;

namespace Pipeline.Services;

public class JsonImportService : IImportService
{
    private static readonly JsonSerializerOptions SerializerOptions = new()
    {
        PropertyNameCaseInsensitive = true
    };

    private readonly string _sourcePath =
        Path.Combine(AppContext.BaseDirectory, "Data", "records.json");

    public List<DataRecord> LoadFromSource()
    {
        if (!File.Exists(_sourcePath))
        {
            throw new FileNotFoundException(
                $"Could not find the source data file at '{_sourcePath}'.",
                _sourcePath);
        }

        string json = File.ReadAllText(_sourcePath);
        List<DataRecord>? records =
            JsonSerializer.Deserialize<List<DataRecord>>(json, SerializerOptions);

        return records ?? [];
    }
}

AppContext.BaseDirectory is the folder the running assembly sits in, so combining it with Data/records.json finds the file no matter where the program was started from. Pair that with the build setting that copies the file next to the output (we'll register that in the project), and the path just works on every machine. It's a small thing that quietly prevents the classic "works on my machine, missing file in production" bug.

Step 2

Two Failures, Handled Differently

One of these problems should stop the program cold. The other absolutely should not. Here's the line between them.

The method guards against two very different kinds of "nothing to load," and treating them the same would be a mistake. A missing file is a broken setup. Someone forgot to ship the data, or the build didn't copy it. The program literally cannot do its job, so we throw a FileNotFoundException with the full path baked into the message. Fail loudly, fail early, and hand whoever's debugging the exact path that didn't exist.

An empty result is a different animal. The file is right there, it parsed fine, it just happened to contain an empty array, or the deserializer handed back null for a body that was literally the text "null". That's not an error; that's a valid file with no records. So records ?? [] turns that null into an empty list and the program carries on with zero records, which is a perfectly reasonable thing to have. The caller never has to null-check the return value, ever.

Crash on a broken contract

A missing file means the program was assembled wrong. There's no graceful way to proceed, so throwing with the offending path is the kindest thing you can do for the next person.

Shrug off an empty one

Zero records is a number, not a failure. Returning an empty list keeps every caller free of null checks and lets the pipeline run end to end with nothing in it.

Step 3

Case-Insensitive Parsing

A tiny options object here buys you forgiveness for the difference between JSON conventions and C# conventions.

That SerializerOptions object exists for one reason, and it's a reason worth knowing. JSON files in the wild tend to use camelCase or lowercase keys (name, category), while C# properties are PascalCase (Name, Category). Turning on PropertyNameCaseInsensitive tells the deserializer to match them regardless of casing, so you're not forced to annotate every single property just to bridge the gap. Combined with the [JsonPropertyName] hints already on the model, the parsing is forgiving without being sloppy.

Notice it's static readonly, built once and shared. Serializer options are reusable and there's a real (if small) cost to constructing them, so there's no reason to make a fresh one on every call. A minor optimization, but it's the kind of detail that signals the code was written with some care.

So this layer reads bytes, deals with both failure modes, and produces clean records. What it pointedly does not do is store anything. Where those records go next is the persistence side, and that starts with the context that knows how to talk to a database.
