Chapter 4

High Level Architecture

The layers of the pipeline, the boundaries they own, and how data flows between them.

The Big Picture

System Map

Click any component to expand its responsibilities and connections.

  • The Composition Root builds and runs the Orchestration Service.
  • The Orchestration Service imports records via the Source Importer and persists them via the Persistence System.
  • The Source Importer produces the Domain Model; the Persistence System stores it.
  • Inside the Persistence System, the Repository delegates to the Persistence Context, which reads and writes the Backing Store.
Boundaries

Dependencies point inward, toward interfaces

No layer references a concrete class in another layer.

The composition root is the only file that knows about concrete types. Everywhere else, a layer accepts its collaborators as interfaces through its constructor. The orchestration service holds an ISourceImporter and an IRepository; it has no idea that one reads a file and the other talks to a database. That ignorance is the whole point — it is what lets you replace either side.

Rule of thumb

If a class uses the new keyword to build one of its collaborators, that collaborator can no longer be swapped or tested in isolation. Construct collaborators in the composition root instead.

Component

Composition Root

The single startup file. It creates the container, registers each interface against its implementation, builds the provider, and resolves the orchestration service to begin the run. It is the only place where concrete classes are named.

Connects to: the container (registration) and the orchestration service (the one object it resolves and invokes).

Component

Orchestration Service

The coordinator. It asks the importer for records, hands them to the repository to persist, reads them back, and presents them. It depends only on the importer and repository interfaces.

Connects to: Source Importer and Repository, both as interfaces.

Component

Source Importer

The inbound adapter. It locates the external source, reads it, and deserializes the contents into a list of domain models. It owns the boundary between the outside world and the pipeline.

Connects to: the Domain Model (produces it) and the external source (reads it).

Component

Repository

The persistence boundary. It exposes a handful of intention-revealing methods — add, get all, count — and hides the storage mechanism entirely behind its interface.

Connects to: the Persistence Context (delegates to it) and the Domain Model (stores and returns it).

Component

Persistence Context

The mapping layer. It declares which models are stored, configures their keys and required fields, and exposes them as a queryable set backed by the chosen provider.

Connects to: the backing store (in-memory or a real database) and the Domain Model (maps it).

Component

Domain Model

The shared currency of the pipeline. A plain data class with a few properties and annotations that describe how it is serialized and stored. Every other layer speaks in terms of this type.

Connects to: every layer — it is produced, persisted, retrieved, and presented.

System

Persistence System

In the main map the storage layers are collapsed into a single node so the diagram stays readable. Inside, a repository delegates to a mapping context, which reads and writes the underlying store.

Component

Backing Store

The concrete storage the context talks to. The pattern keeps it swappable: an in-memory provider for tests and local runs, a real database in production. Nothing above this layer changes when it is replaced.
