High Level Architecture

The Big Picture

System Map

  • The Composition Root builds and starts the Orchestration Service.
  • The Orchestration Service loads records via the Import Service.
  • The Orchestration Service stores and reads records via the Persistence System.
  • The Import Service reads the JSON Source file.

Reading the Map

How Data Moves Through the Layers

There's one trip out to fetch data and one trip back to show it, and the arrows tell you who is allowed to talk to whom.

Trace a single run left to right. The Composition Root is the only box that touches every other box, and it does its job once: it registers the layers, builds the object graph, and asks for the Orchestration Service. From that point on, the root is done and the conductor takes over.

The Orchestration Service reaches two ways. It asks the Import Service to load records, and the Import Service in turn reads the JSON Source off disk and parses it. Then the orchestrator hands those records to the Persistence System to store, and asks for them back to print. Notice what's missing from the diagram: there's no arrow from the Orchestration Service straight to the JSON Source, and none from the Import Service into the database. Each layer reaches exactly one step further and no more. That restraint is the architecture.

Dependencies point one way

Arrows flow from caller to callee and never loop back. The orchestrator knows about the import and persistence layers; neither of them has ever heard of the orchestrator. That one-way rule is what keeps changes contained.

The database hides inside a system

Three persistence pieces are folded into one node so the top-level map stays a clean line. Open it to see the Repository, the Persistence Context, and the Domain Model that live behind that single boundary.

That collapsed Persistence System is worth opening now, because the next several chapters live inside it. Click it in the map above and you'll see the same one-way flow at a smaller scale. Then we start building, beginning with the one piece every arrow eventually carries: the Domain Model.

Component

Composition Root

The startup block that owns assembly. It registers each interface against a concrete implementation, registers the Persistence Context against a database provider, builds the container, then resolves and runs the Orchestration Service. It's the one place in the program that names concrete types, which is exactly why nothing else has to.

Component

Orchestration Service

The conductor. It depends on the Import Service and the Repository through their interfaces and does no parsing or persistence itself. Its single responsibility is sequence: load, store, retrieve, present. Because it only knows abstractions, it's the easiest layer in the system to test.

Component

Import Service

The boundary to the outside world. It locates the source file, guards against it being missing, reads the bytes, and deserializes them into Domain Model records. Everything fragile about external input, paths, file existence, malformed data, is contained here so the rest of the pipeline can assume clean records.

External

JSON Source

The raw input: a JSON file copied next to the binaries at build time. It sits outside the program's own types, which is why the Import Service exists to translate it. Swap this for an HTTP endpoint or a CSV file and only the Import Service changes.

System

Persistence System

Collapsed in the main map to keep the diagram a straight line. Inside, a thin Repository talks to a Persistence Context, which maps the Domain Model onto database rows. The Repository is the only part the outside world ever names; the context and the model stay tucked behind it.

Component

Repository

A small interface, three or four methods, that lets callers add and fetch records without knowing a database is involved. It owns the change-tracking and save calls so nobody above it has to. This is the seam where you could replace the entire storage engine and break nothing upstream.

Component

Persistence Context

The Entity Framework Core context. It exposes the set of records, configures how the Domain Model maps to a table, and receives its database provider from the outside through its options. Choosing in-memory versus a real database happens at registration, not in this class.

Component

Domain Model

A plain class with a handful of properties and no behavior. It carries a key plus serialization hints so it can double as both the JSON shape and the database row. Every other component passes instances of it around; it's the shared vocabulary of the whole pipeline.
