Chapter 5

Domain Model

We build from the inside out, and the innermost piece is the record everything else carries around.

Step 1

A Class With No Behavior

Start with the data, because every other layer is defined by what it does to this thing.

The architecture map ended with the Domain Model sitting deep inside the persistence system, and that's a good place to start building. It's the smallest piece and the one every arrow eventually carries. Get it right and the layers above it have something solid to stand on.

Here's the whole thing. A handful of properties and not a single method:

using System.ComponentModel.DataAnnotations;
using System.Text.Json.Serialization;

namespace Pipeline.Models;

public class DataRecord
{
    [Key]
    [JsonPropertyName("id")]
    public Guid Id { get; set; }

    [JsonPropertyName("name")]
    public string Name { get; set; } = string.Empty;

    [JsonPropertyName("category")]
    public string Category { get; set; } = string.Empty;

    [JsonPropertyName("quantity")]
    public int Quantity { get; set; }
}

That's deliberate. The Domain Model is sometimes called an anemic model, and people say that like it's a criticism, but here it's the entire point. This class doesn't validate itself, doesn't talk to a database, doesn't know what JSON is. It's a bag of fields that means the same thing in every layer. When the Import Service produces records and the Repository stores them and the Orchestration Service prints them, they're all holding the same shape, and none of them disagrees about what a record is.

Step 2

One Class, Two Jobs

Those attributes aren't decoration. Each one answers a different system that needs something from this type.

Look again at the attributes, because they're doing real work for two different consumers. The [JsonPropertyName] attributes belong to the serializer: they map the lowercase keys in the source file (name, category) onto the C# properties (Name, Category). The [Key] attribute belongs to the ORM: it tells Entity Framework Core which property is the primary key when this record becomes a database row.

So one plain class quietly serves both the import side and the persistence side. That's a trade-off worth being honest about. The clean-architecture purist would split this into a separate JSON DTO and a separate database entity, keeping each free of the other's attributes. They'd have a point. But for a pipeline this size, two parallel classes plus mapping code between them is more machinery than the problem deserves. We let one type wear both hats and keep the field definitions in a single place.

When to split it later

The moment the JSON shape and the stored shape start drifting apart, say the source sends a full address but you only persist a city, that's your signal to break this into two types with a mapping step. Until then, one class is the simpler honest choice.

Step 3

Sensible Defaults and the Key Type

Two small decisions in here save you from a category of null bugs and id collisions.

Notice = string.Empty on the two string properties. With nullable reference types switched on, the compiler would otherwise nag that a non-null string might never get assigned. Defaulting to empty makes the promise true: a freshly constructed record has real empty strings, not nulls waiting to throw somewhere downstream. It's a one-liner that removes a whole class of "object reference not set" surprises later.

The key is a Guid rather than an auto-incrementing integer, and that's a choice with consequences. A Guid can be generated anywhere, by the source system, by your importer, by a test, without coordinating with the database to find the next free number. For a pipeline that ingests records from outside, that independence matters: the id arrives already baked into the JSON, and you store it as-is instead of letting the database mint a new one. The cost is wider keys and ids that aren't human-friendly, which is a fine trade when records are born outside your database.

That's the entire foundation. With a record type that both the serializer and the ORM understand, we can build the layer that brings these records into the program in the first place: the Import Service.
