Chapter 10

The Composition Root

Assemble every layer in one place, let the container resolve the graph, and run the pipeline.

Concept

One place that knows the concrete types

Every other file depends on interfaces; this one names implementations.

The composition root is the single location where the abstract and the concrete meet. It maps each interface to the class that implements it, then builds a provider that can hand out fully-constructed objects on demand. Concentrating this knowledge in one file is what keeps the rest of the codebase free of wiring.

Step 1

Register the layers

Bind each interface to its implementation, and configure the context's provider.

Create the container and register every layer. The persistence context is configured here with the in-memory provider — the one place that decides where data is stored. Each interface is bound to its concrete class with a scoped lifetime.

ServiceCollection services = new();

services.AddDbContext<PersistenceContext>(options =>
    options.UseInMemoryDatabase("PipelineDb"));

services.AddScoped<IRepository, Repository>();
services.AddScoped<ISourceImporter, SourceImporter>();
services.AddScoped<IOrchestrationService, OrchestrationService>();
Step 2

Build the provider and prepare the store

Resolve the context once to create a clean, deterministic database.

Build the provider, then reset the store so every run starts from a known state. Resolving the context from the provider — rather than constructing it — means it arrives already configured with the provider you registered.

using ServiceProvider provider = services.BuildServiceProvider();

PersistenceContext context = provider.GetRequiredService<PersistenceContext>();
context.Database.EnsureDeleted();
context.Database.EnsureCreated();
Step 3

Resolve the top of the graph and run

Ask for one object; the container constructs everything beneath it.

Request the orchestration service. The container sees it needs an importer and a repository, that the repository needs the context, and constructs the whole graph for you. Then call the single entry method to run the pipeline.

IOrchestrationService service =
    provider.GetRequiredService<IOrchestrationService>();

service.ImportAndDisplay();
The payoff

You asked for one object and received a fully-wired pipeline. To change where data comes from or where it is stored, you edit only the registrations in this file — never the layers themselves. That is the entire promise of the pattern, delivered in a dozen lines.

Step 4

The full entry point

Every piece, assembled top to bottom.

ServiceCollection services = new();

services.AddDbContext<PersistenceContext>(options =>
    options.UseInMemoryDatabase("PipelineDb"));

services.AddScoped<IRepository, Repository>();
services.AddScoped<ISourceImporter, SourceImporter>();
services.AddScoped<IOrchestrationService, OrchestrationService>();

using ServiceProvider provider = services.BuildServiceProvider();

PersistenceContext context = provider.GetRequiredService<PersistenceContext>();
context.Database.EnsureDeleted();
context.Database.EnsureCreated();

IOrchestrationService service =
    provider.GetRequiredService<IOrchestrationService>();

service.ImportAndDisplay();

Run it with dotnet run. The records flow from source to store to screen — and you have built the complete pattern, one swappable layer at a time.
