Chapter 6

The Domain Model

Define the single record that every layer produces, stores, retrieves, and presents.

Why First

Start with the data

The model is the shared vocabulary the whole pipeline speaks.

Every layer in this pattern passes the same type around: the importer produces it, the repository stores it, the orchestration service presents it. Defining that type first gives the rest of the pipeline a stable shape to build against. Keep it plain — a handful of properties with no behavior.

Step 1

Declare the properties

A unique key plus the fields you care about.

The model needs a stable identifier and the data fields it carries. Use a globally unique identifier for the key so records never collide regardless of where they came from.

public class DataRecord
{
    public Guid Id { get; set; }

    public string Name { get; set; } = string.Empty;

    public string Label { get; set; } = string.Empty;

    public int Amount { get; set; }
}
Step 2

Annotate for serialization and storage

Two sets of annotations let one class serve two layers.

Because this same class is both deserialized from the source and stored by the persistence context, you annotate it for both jobs. Serialization attributes map external field names onto properties; the persistence attribute marks the key. The class stays a plain data holder — the annotations only describe it.

using System.ComponentModel.DataAnnotations;
using System.Text.Json.Serialization;

public class DataRecord
{
    [Key]
    [JsonPropertyName("id")]
    public Guid Id { get; set; }

    [JsonPropertyName("name")]
    public string Name { get; set; } = string.Empty;

    [JsonPropertyName("label")]
    public string Label { get; set; } = string.Empty;

    [JsonPropertyName("amount")]
    public int Amount { get; set; }
}
Design note

Sharing one class between the source and the store is convenient for a small pipeline. As a system grows, many teams split this into a separate transport model and storage entity. The pattern accommodates both — only the importer and context would change.

Takeaway

One type, every layer

The model is the contract the rest of the guide builds on.

With the record defined, each remaining layer has something concrete to accept and return. Next you build the layer that brings these records into the pipeline from the outside world.
