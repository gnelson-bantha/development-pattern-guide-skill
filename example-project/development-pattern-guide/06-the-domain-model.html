Chapter 6

The Domain Model

The entity that flows through every layer, why it stays free of behavior, and how attributes connect it to storage and to the external format.

Definition

A Plain Data Object

The shape of the entity and the fields it carries.

The domain model is the single data shape every other layer agrees on. The Import Service produces it, the Persistence System stores it, and the Orchestration Service reads it. Because it is shared by everything, it is kept deliberately small: fields and nothing else.

using System.ComponentModel.DataAnnotations;
using System.Text.Json.Serialization;

public class Entity
{
    [Key]
    [JsonPropertyName("id")]
    public Guid Id { get; set; }

    [JsonPropertyName("name")]
    public string Name { get; set; } = string.Empty;

    [JsonPropertyName("category")]
    public string Category { get; set; } = string.Empty;

    [JsonPropertyName("value")]
    public int Value { get; set; }
}
Design Choice

No Behavior on the Entity

Why the model holds data and the services hold logic.

The entity has properties and no methods. This is a choice, not an oversight. If the model carried persistence logic it would depend on the ORM, and every layer that touches the model would transitively depend on the ORM too, which defeats the separation the pattern exists to create. If it carried import logic it would depend on the serializer. Keeping the model inert means it can be referenced everywhere without dragging a dependency along with it.

The trade-off is that behavior has to live somewhere, and here it lives in the services. This is the data-centric end of the design spectrum: the model is a transport record and the layers around it own the rules. For a richer domain you might push invariants onto the model, but you would then accept that the model is no longer dependency-free. For this pattern, the inert model is what lets the same type cross every boundary cleanly.

Attributes

Two Mappings on One Type

How the entity serves both the store and the external format at once.

The model is annotated for two different systems. The [Key] attribute tells the ORM which property is the primary key. The [JsonPropertyName] attributes tell the serializer how each property maps to a field in the external source. One type carries both contracts.

This is convenient and worth understanding as a deliberate compromise. The model is coupled to two libraries through attributes, which keeps the example small. The cost is that the same class now answers to both the database schema and the wire format, so a change to either can ripple into the other. If those two concerns diverge, the clean move is to split into a separate persistence entity and a separate transfer object and map between them in the Import Service. Until they diverge, one annotated model is the smaller, clearer choice.

Key Attribute

Marks the identity property so the ORM can track, update, and de-duplicate the entity in the store.

Property Name Attribute

Binds an external field name to a property so the source format and the C# naming can differ without custom parsing.

Default Initializers

String properties default to empty rather than null, so a partially populated source never leaves a property in an invalid null state.
