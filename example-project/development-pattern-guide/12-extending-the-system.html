Chapter 12

Extending the System

How the pattern grows: swapping an implementation, adding a new entity, and knowing when a layer is no longer worth its weight.

Substitution

Replacing an Implementation

Changing how a layer works without touching its callers.

The most common extension is swapping one implementation for another. Because every caller depends on an interface, you add a class that satisfies the interface and change the registration in the composition root. Moving from a file source to an HTTP source is a new class and a one-line registration change:

public class HttpImportService(HttpClient client) : IImportService
{
    public List<Entity> LoadFromSource()
    {
        string json = client.GetStringAsync("https://example/data").Result;
        return JsonSerializer.Deserialize<List<Entity>>(json) ?? [];
    }
}
// In the composition root, swap one line:
services.AddScoped<IImportService, HttpImportService>();

The Orchestration Service is not recompiled in spirit and not edited at all, because it only ever knew IImportService. The same move replaces the in-memory database with a real one by changing the context options, and the repository above it is untouched. Substitution is the dividend the interfaces have been paying into the whole guide.

New Entities

Adding a Second Entity Type

Repeating the layer set for a new kind of data.

To persist a second kind of data, repeat the pattern for it: a new model, a new entity set on the context, and a repository contract for it. The existing layers are not disturbed, because each new piece plugs into the same seams.

public class OtherEntity
{
    public Guid Id { get; set; }
    public string Label { get; set; } = string.Empty;
}

// On the context:
public DbSet<OtherEntity> OtherEntities => Set<OtherEntity>();

If you find yourself copying the repository almost verbatim for each new entity, that is the signal to introduce a generic repository (IRepository<T>) so one implementation serves every entity type. Reach for that generalization when the duplication is real, not before: a generic abstraction written against a single entity is harder to read and earns nothing. The pattern tolerates both the per-entity and the generic form; let the amount of repetition decide.

Judgment

Knowing When a Layer Earns Its Keep

Adding seams where substitution is plausible and removing them where it is not.

Every layer in this pattern is justified by a substitution that is plausible: a different data source, a different store, a fake for testing. A layer that guards a substitution no one will ever make is pure overhead, and the honest move is to delete it and call the concrete class directly. The pattern is a tool for managing change, so apply it where change is expected and skip it where it is not.

When you do add a layer, hold it to the standards the existing ones meet: define the interface first, keep it expressed in domain terms, let the composition root own construction and lifetimes, and validate input at the boundary where it enters. A new layer that follows those rules slots in without friction. A new layer that breaks them, by leaking the ORM through the repository or constructing its own dependencies, quietly erodes the separation the rest of the system depends on.

Add at the Seam

New behavior enters by implementing an existing interface or adding a new one, never by editing an unrelated layer.

Register, Do Not Construct

Anything new is wired in the composition root, keeping the rest of the program free of concrete type names.

Delete Dead Seams

A layer protecting a substitution that will never happen is cost without benefit; removing it is as valid as adding one.
