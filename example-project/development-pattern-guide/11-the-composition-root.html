Chapter 11

The Composition Root

The one place that names concrete types: registering each contract, choosing lifetimes, building the provider, and starting the run.

The Root

Registration, Build, Run

The whole entry point, top to bottom.

Every other layer was written against interfaces and constructed nothing. That deferral has to end somewhere: a real program needs real objects. The composition root is that somewhere. It is the single component that knows the concrete classes, maps each interface to its implementation, and hands the finished object graph to the runtime.

using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.DependencyInjection;

ServiceCollection services = new();

services.AddDbContext<DataContext>(options =>
    options.UseInMemoryDatabase("AppDb"));

services.AddScoped<IRepository, Repository>();
services.AddScoped<IImportService, ImportService>();
services.AddScoped<IOrchestrationService, OrchestrationService>();

using ServiceProvider provider = services.BuildServiceProvider();

DataContext context = provider.GetRequiredService<DataContext>();
context.Database.EnsureDeleted();
context.Database.EnsureCreated();

IOrchestrationService service = provider.GetRequiredService<IOrchestrationService>();
service.Run();
Why One Place

Concentrating Construction

The reason every concrete type is named here and nowhere else.

The registrations map each interface to a class: IRepository to Repository, IImportService to ImportService, IOrchestrationService to OrchestrationService. This is the only file in the program where the concrete names appear together. Because construction is concentrated here, the shape of the whole system is visible in one screen, and changing an implementation is a one-line edit that no other layer notices.

When the provider is asked for IOrchestrationService, it sees that the service needs an IImportService and an IRepository, that the repository needs a DataContext, and that the context needs its options. It constructs the entire graph from the leaves up and injects each object into the one above it. You wrote constructors that declare what each layer needs; the container reads those declarations and satisfies them. No layer ever calls new on a collaborator.

Lifetimes and Startup

Scopes, the Provider, and First Run

Choosing how long objects live and preparing the store before use.

Each registration carries a lifetime. AddScoped ties an object's life to a scope, which matters most for the database context: a context is a unit of work and should not be shared for the life of the process, so a scoped lifetime gives each logical operation its own. The choice of lifetime is part of the wiring, and it belongs in the root alongside the registrations rather than being decided inside the classes.

After building the provider, the root prepares the store before resolving the service. The EnsureDeleted then EnsureCreated pair gives a clean store on every run, which suits a deterministic example; a real application would replace this with migrations. The using on the provider ensures it is disposed when the program ends, which disposes the context and releases anything it held. Startup preparation and shutdown both live in the root because both are part of assembling and tearing down the system, which is the root's job.

Interface to Class

Each registration binds a contract to one implementation, so swapping implementations means changing only this file.

Scoped Context

The database context is scoped so each unit of work gets a fresh, short-lived context rather than one shared across the whole process.

Resolve the Top, Not the Parts

The root asks the provider only for the top-level service; the container constructs and injects everything beneath it automatically.
