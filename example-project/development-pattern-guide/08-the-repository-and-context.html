Chapter 8

The Repository & Context

Hide storage behind a focused interface, backed by a mapping context the rest of the pipeline never sees.

Step 1

Describe storage as intentions

The interface names what callers want, not how it happens.

The repository interface exposes only the operations the pipeline actually needs: add a batch of records, read them all back, and count them. None of these methods reveal the storage technology, which keeps callers decoupled from it.

public interface IRepository
{
    void AddRange(IEnumerable<DataRecord> records);

    List<DataRecord> GetAll();

    int Count();
}
Step 2

Define the persistence context

The context maps the domain model onto a backing store.

The context declares which models are stored and configures how they are mapped — the key and any required fields. It receives its options through the constructor, so the composition root decides which provider backs it.

public class PersistenceContext(DbContextOptions<PersistenceContext> options)
    : DbContext(options)
{
    public DbSet<DataRecord> Records => Set<DataRecord>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<DataRecord>(entity =>
        {
            entity.HasKey(record => record.Id);
            entity.Property(record => record.Name).IsRequired();
            entity.Property(record => record.Label).IsRequired();
        });
    }
}
Step 3

Implement the repository over the context

A thin class that delegates to the context and nothing more.

The concrete repository receives the context through its constructor and translates each interface method into a context operation. It stays deliberately thin — the context does the real work, and the interface keeps that fact hidden from callers.

public class Repository(PersistenceContext context) : IRepository
{
    public void AddRange(IEnumerable<DataRecord> records)
    {
        context.Records.AddRange(records);
        context.SaveChanges();
    }

    public List<DataRecord> GetAll() => [.. context.Records];

    public int Count() => context.Records.Count();
}
Two layers, one boundary

The context is an implementation detail of the repository. Callers depend on IRepository only, so you can later replace the in-memory provider with a real database — or even drop the ORM entirely — without changing a single caller.

Takeaway

Persistence, sealed off

The storage decision now lives behind one interface.

With the importer producing records and the repository storing them, only one piece remains: the layer that drives the sequence. That is the orchestration service, next.
