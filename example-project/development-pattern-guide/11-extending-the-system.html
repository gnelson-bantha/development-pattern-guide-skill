Chapter 11

Extending the System

Grow the pipeline by adding implementations behind the existing interfaces — never by editing the layers that depend on them.

Where to Hook In

Extend at the interfaces

Each abstraction is a seam where new behavior can be added.

The pattern was built so that growth happens at the seams. Every layer already hides behind an interface, and the composition root is the only place that chooses implementations. To extend the system you write a new class and change one registration — nothing downstream needs to know.

▸

New source

Add an ISourceImporter that reads a network API or a different format.

▦

New store

Swap the repository's provider — or its whole implementation — for a real database.

◆

New behavior

Wrap a layer with a decorator that adds logging, caching, or validation.

Example 1

Add a second source

A new importer that pulls records from a remote service.

Implement the same interface with a different strategy. Callers continue to depend on ISourceImporter, so the orchestration service is unaffected. Only the registration changes.

public class RemoteSourceImporter(HttpClient client) : ISourceImporter
{
    public List<DataRecord> LoadFromSource()
    {
        string json = client.GetStringAsync("/records").Result;
        return JsonSerializer.Deserialize<List<DataRecord>>(json) ?? [];
    }
}

// In the composition root, swap one line:
services.AddScoped<ISourceImporter, RemoteSourceImporter>();
Example 2

Move to a real database

Change the provider without touching the repository.

Because the persistence context receives its options from the composition root, switching from the in-memory provider to a durable one is a registration change. The repository and every caller stay exactly the same.

// Replace the in-memory registration with a real provider:
services.AddDbContext<PersistenceContext>(options =>
    options.UseSqlite("Data Source=pipeline.db"));
Migrations

A durable store replaces EnsureCreated with schema migrations. The composition root applies them at startup, but the layers above remain untouched — they only ever see IRepository.

Example 3

Add cross-cutting behavior with a decorator

Layer new responsibilities without modifying existing classes.

A decorator implements an interface and wraps another instance of it. This lets you add concerns such as logging or caching around any layer while leaving the original class and its callers untouched.

public class LoggingRepository(IRepository inner) : IRepository
{
    public void AddRange(IEnumerable<DataRecord> records)
    {
        Console.WriteLine("Persisting a batch of records...");
        inner.AddRange(records);
    }

    public List<DataRecord> GetAll() => inner.GetAll();

    public int Count() => inner.Count();
}
Guardrails

Keep extensions honest

A few habits preserve the pattern as the system grows.

  • Add behavior by implementing interfaces, not by editing the orchestration service.
  • Keep validation at the edges — in importers — so inner layers stay simple.
  • Let the composition root remain the single place that names concrete types.
  • If an interface starts to grow many unrelated methods, split it so each caller depends only on what it uses.
The pattern, restated

A one-way flow of data through small layers, each hidden behind an interface and assembled in one place. Hold to that, and the system stays easy to extend no matter how large it becomes.
