Chapter 11

Extending the System

A design is only as good as the changes it survives. Here's how new sources, stores, and steps slot in without disturbing what's already there.

New Inputs

Swapping the Data Source

The day the data stops living in a file, you'll be glad the orchestrator never knew it was a file to begin with.

Say the records move from a local JSON file to a REST API. In a tangled program that's a scary change, because file-reading code is probably braided through everything. Here it's almost boring. Write a new class that implements the same IImportService contract:

public class HttpImportService(HttpClient client) : IImportService
{
    public List<DataRecord> LoadFromSource()
    {
        string json = client.GetStringAsync("https://example/records").Result;
        return JsonSerializer.Deserialize<List<DataRecord>>(json) ?? [];
    }
}

Then change one line in the composition root from JsonImportService to HttpImportService. That's the whole edit. The orchestrator still asks for an IImportService and still gets back a list of records; it has no idea the bytes now come over the wire instead of off the disk. This is the import boundary doing exactly the job it was drawn for. The contract said "give me records" and stayed silent about where from, and that silence is what makes the swap a one-liner.

New Storage

Changing Where Records Live

Two different changes hide here: switching the database engine, and replacing storage entirely. The pattern handles both at different seams.

There are two flavors of "store it somewhere else," and the design gives you a different seam for each. If you just want a real database instead of the in-memory one, you don't touch the repository at all. The provider is chosen in the composition root, so you change UseInMemoryDatabase to UseSqlite (or SQL Server, or Postgres), add the matching EF Core package, and you're done. The repository, the context class, and everything above them stay exactly as written.

If you want to leave EF Core behind entirely, maybe records belong in a cloud document store, the IRepository interface is your seam. Implement those same three methods against the new backend and register it instead:

public class DocumentRepository(DocumentClient client) : IRepository
{
    public void AddRange(IEnumerable<DataRecord> records) => client.InsertMany(records);
    public List<DataRecord> GetAll() => client.Query<DataRecord>().ToList();
    public int Count() => client.Query<DataRecord>().Count();
}

Same engine, real database

Switch the provider line in the composition root and add the package. The repository never changes, because choosing the provider was always the root's job, not the repository's.

Different engine entirely

Implement IRepository against the new store and register it. Because that interface is the only thing the orchestrator ever saw, nothing above the seam reacts.

New Steps

Adding Stages to the Pipeline

When the process itself grows a step, there's exactly one place that should know about it.

What if the work itself changes? Suppose you need to validate or deduplicate records after loading but before storing. The instinct might be to wedge that logic into the import service, but resist: importing is about reading the source, not about business rules. A new step is a new concern, so give it its own interface and class, say an IRecordValidator, and inject it into the orchestrator alongside the others.

The orchestrator is the right home because the orchestrator already owns sequence. Adding a step means adding a line between "load" and "store," and that line belongs where all the other ordering lives:

public void ImportAndDisplay()
{
    List<DataRecord> imported = importService.LoadFromSource();
    List<DataRecord> valid = validator.Filter(imported);   // the new step
    repository.AddRange(valid);
    // ... read back and present as before
}

One new class, one new registration, one new line in the conductor. The import service still just imports, the repository still just stores, and the new rule lives in a class you can test on its own. That's the test of a layered design, and this one passes: each kind of change has an obvious, contained home, and none of them forces you to reopen code that already works.
