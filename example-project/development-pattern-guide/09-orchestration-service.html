Chapter 9

Orchestration Service

Every worker is built. This layer is the conductor that tells them when to play, and it does almost nothing itself.

Step 1

Depending on Two Interfaces

The conductor needs the import and persistence layers, but it asks for the abstractions, never the concrete classes.

We've got a layer that loads records and a layer that stores them, but so far nothing tells them to actually do it in sequence. That's this layer's whole job. Contract first, as usual, and it's almost comically simple:

namespace Pipeline.Services;

public interface IOrchestrationService
{
    void ImportAndDisplay();
}

One method. The orchestrator presents a single verb to the outside world, "run the whole thing," and hides every step behind it. Now the implementation, and the constructor is where the pattern pays off:

using Pipeline.Models;
using Pipeline.Repositories;

namespace Pipeline.Services;

public class OrchestrationService(
    IImportService importService,
    IRepository repository) : IOrchestrationService
{
    public void ImportAndDisplay()
    {
        List<DataRecord> imported = importService.LoadFromSource();
        repository.AddRange(imported);

        List<DataRecord> stored = repository.GetAll();

        Console.WriteLine($"Loaded {repository.Count()} records from the store:");
        Console.WriteLine();

        foreach (DataRecord record in stored)
        {
            Console.WriteLine($"Id:       {record.Id}");
            Console.WriteLine($"Name:     {record.Name}");
            Console.WriteLine($"Category: {record.Category}");
            Console.WriteLine($"Quantity: {record.Quantity:N0}");
            Console.WriteLine(new string('-', 40));
        }
    }
}

The constructor asks for IImportService and IRepository, the interfaces, not JsonImportService and PipelineRepository, the classes. That's the line that keeps this layer honest. The orchestrator has no idea its records come from JSON or that they land in Entity Framework. It knows only that something can load records and something can store them. Swap the JSON importer for an HTTP one and this file is untouched, because it was never looking at the concrete type in the first place.

Step 2

Sequence, Not Substance

Read what the method actually does and you'll notice it never does any real work, it just decides the order.

Walk through ImportAndDisplay and there's a rhythm to it: load, then store, then read back, then print. Four steps, and the orchestrator owns the order they happen in but none of the work inside them. It doesn't parse a file, it asks the import service to. It doesn't run a query, it asks the repository to. Its real contribution is the sequencing decision: records have to be loaded before they can be stored, and stored before they can be read back.

This is what separates an orchestrator from a layer that does a job. The import service does a job. The repository does a job. The orchestrator coordinates jobs. Keeping coordination in its own layer means the question "what happens, and in what order?" has a single answer in a single file. When the business process changes, say you need to deduplicate records between loading and storing, you know exactly where that logic goes, and you know it won't disturb how files are read or how the database works.

Read-back over trusting the input

It prints the records fetched from the repository, not the list it just imported. Asking the store for the truth confirms the round trip actually worked instead of assuming it did.

Presentation lives at the top

The console formatting sits here, in the outermost layer, so the workers below stay free of any opinion about how results get shown. Swap console for a file writer and only this layer moves.

Step 3

The Layer That's Trivial to Test

Because it touches nothing real, you can prove its logic without a file or a database anywhere in sight.

Here's the quiet reward for all this discipline. The orchestrator depends only on two interfaces, so a test can hand it fakes and check the sequence without any infrastructure at all. The nice part is testing. Swap in a fake repository and the layers above don't notice the difference:

var fakeImport = new StubImportService(threeRecords);
var fakeRepo = new InMemoryRepository();

var orchestrator = new OrchestrationService(fakeImport, fakeRepo);
orchestrator.ImportAndDisplay();

// Did the orchestrator actually store what was imported?
Assert.Equal(3, fakeRepo.Count());

No file on disk, no EF Core, no setup or teardown. The test verifies the thing the orchestrator is actually responsible for, that imported records make it into the store, and it runs in microseconds. This is the concrete answer to "why bother with all these interfaces?" The ceremony from earlier chapters cashes out right here, as the ability to test the most important logic in the program in isolation.

So every layer exists, and every layer talks only to abstractions. But somebody still has to decide that IRepository means PipelineRepository and hand the orchestrator its real collaborators. Nothing we've written does that yet. That final job belongs to the composition root.
