Chapter 10

Composition Root

Every layer has politely refused to name the next one. Here, finally, is the one place that names them all and snaps the graph together.

Step 1

Registering the Layers

This is where each interface is told which concrete class it really means, all in one readable block.

Throughout the build, every class asked for interfaces and constructed nothing. That left a gap: somebody has to decide that IRepository means PipelineRepository. The composition root is that somebody, and it lives in Program.cs as the program's first executing code:

using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.DependencyInjection;
using Pipeline.Data;
using Pipeline.Repositories;
using Pipeline.Services;

ServiceCollection services = new();

services.AddDbContext<PipelineDbContext>(options =>
    options.UseInMemoryDatabase("PipelineDb"));

services.AddScoped<IRepository, PipelineRepository>();
services.AddScoped<IImportService, JsonImportService>();
services.AddScoped<IOrchestrationService, OrchestrationService>();

Each AddScoped line is a sentence: "when something asks for this interface, build this class." Read the three of them and you've read the entire map of which abstraction resolves to which implementation. This is the one and only file in the program that names both sides of every pairing, and that's exactly why every other file got to stay ignorant. The knowledge about concrete types didn't disappear; it got concentrated here, where you can see all of it at a glance.

The AddDbContext line is where the database finally gets chosen. Remember the context took its provider through the constructor and refused to pick one itself? This is the moment that choice gets made. We pass UseInMemoryDatabase, and changing this single line to UseSqlite or UseSqlServer would move the whole program onto a real database without touching another file. The seam the context left open gets filled right here.

Step 2

Why Scoped, and What a Lifetime Is

That word "scoped" is a real decision about how long objects live and who shares them.

Every registration says AddScoped, and that word picks a lifetime. The container offers three. Singleton means one instance for the whole application. Transient means a brand-new instance every single time someone asks. Scoped sits in between: one instance per scope, where a scope is usually one unit of work, like a single request or, here, a single run.

Scoped is the right call for anything that touches a database. EF Core's context isn't built to be shared across concurrent work, and it accumulates change-tracking state as you use it. You want a fresh one per unit of work and you want everyone in that unit to share the same one, so the repository and the context line up on the same change tracker. Scoped gives you exactly that. Pick singleton instead and you'd be sharing one context forever, which leaks state and breaks under concurrency. Pick transient and the repository might get a different context than expected. The lifetime isn't a detail you sprinkle on; it's a correctness decision, and scoped is the one that matches a database-backed unit of work.

The failure mode to remember

Most dependency-injection bugs are lifetime mismatches: a long-lived object holding a short-lived one, or two collaborators that were supposed to share a context ending up with two different ones. When something behaves strangely, check the lifetimes first.

Step 3

Building the Graph and Pulling the Trigger

One resolve call quietly constructs the entire object tree you spent six chapters building.

Registration just describes the parts. Now we build the container and run:

using ServiceProvider provider = services.BuildServiceProvider();

// Start each run from a clean, freshly created store.
PipelineDbContext dbContext = provider.GetRequiredService<PipelineDbContext>();
dbContext.Database.EnsureDeleted();
dbContext.Database.EnsureCreated();

IOrchestrationService orchestrator =
    provider.GetRequiredService<IOrchestrationService>();
orchestrator.ImportAndDisplay();

That single GetRequiredService<IOrchestrationService>() call does something genuinely satisfying. The container sees the orchestrator needs an IImportService and an IRepository, sees the repository needs a PipelineDbContext, sees the context needs its options, and builds the whole tree from the bottom up, handing each constructor exactly what it asked for. You wrote a dozen "give me what I need" constructors across six chapters, and this one line satisfies all of them at once. That's dependency injection delivering on its promise.

The EnsureDeleted then EnsureCreated pair resets the in-memory store so every run starts empty and produces the same output, which keeps the demo honest. The using on the provider matters too: when the program ends, disposing the provider disposes the scoped services it created, including the context, so the database connection is cleaned up properly. Even teardown follows the rule that the composition root owns the lifecycle.

That completes the pattern. Six small layers, each behind an interface, each ignorant of the others, all assembled in one readable block. From here, the real test of a design is how it grows, so the next chapter walks through adding new sources, stores, and steps without disturbing what you've built.
