Chapter 7

The Database Context

The ORM's view of the store: which entities it persists, how they map to tables, and why its options arrive from outside.

Definition

The Context and Its Entity Set

Declaring what is persisted.

The database context is the ORM's session with the store. It exposes one entity set per persisted type and, when asked, translates operations on those sets into store commands. For a single entity it is short:

using Microsoft.EntityFrameworkCore;

public class DataContext(DbContextOptions<DataContext> options) : DbContext(options)
{
    public DbSet<Entity> Entities => Set<Entity>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Entity>(entity =>
        {
            entity.HasKey(e => e.Id);
            entity.Property(e => e.Name).IsRequired();
            entity.Property(e => e.Category).IsRequired();
        });
    }
}
Injection

Options Passed Through the Constructor

Why the context does not choose its own provider.

The constructor takes a DbContextOptions and passes it to the base class. The context never decides which database provider it uses or what the connection looks like. That decision is made in the Composition Root and handed in. This is the same dependency injection rule the rest of the pattern follows, applied to the persistence layer: the context receives its configuration rather than constructing it.

The payoff is direct. The same context class runs against an in-memory provider in a test and a real database in production, because the provider is in the options, not in the class. Nothing in the context changes when the store changes. If the context picked its own provider, you would have to edit it for every environment, and you could not point a test at a throwaway store without touching production code.

Configuration

Model Configuration in One Place

Centralizing the mapping rules the store must enforce.

The OnModelCreating override is where mapping rules that the store should enforce live: the primary key, which columns are required, and any constraints. The entity already carries a [Key] attribute, so the explicit HasKey here is redundant for the key itself; it is shown to make the point that the context is the single place to put configuration the attributes cannot express, and to keep schema rules out of the model when you prefer them gathered together.

There are two valid styles: attributes on the model, or fluent configuration in the context. Attributes keep the rule next to the property; fluent configuration keeps all rules in one file and avoids coupling the model to the ORM. The pattern tolerates either. The thing to avoid is scattering the same rule across both, where the two can disagree.

Entity Set

Each DbSet is a queryable, trackable collection of one entity type. Adding a new persisted type means adding one set.

Change Tracking

The context records the entities it has seen so that a later save knows which rows to insert or update. The repository above it triggers that save.

Provider Independence

Because the provider lives in the injected options, the context's code is identical across in-memory, local, and production stores.
