Pattern Tutorial

Chapter 1

Introduction

A four-line program hides a surprisingly clean pattern. Let's name it before we build it.

The Idea

What We're Actually Building

Strip away the domain and it's the same job a thousand programs do: take data from somewhere, keep it, hand it back.

Here's the whole task in one breath: read a list of records from a file, store them in a database, read them back out, and print them. You could write that in twenty lines inside Main and go home. Plenty of people do, and for a throwaway script that's the right call.

But watch what happens when the script stops being throwaway. The file becomes an HTTP endpoint. The database picks up a second table. Someone asks for the records sorted, or filtered, or exported as CSV instead of printed. Every one of those changes lands in the same giant Main, and they all touch each other. Parsing breaks persistence; persistence breaks formatting. You're editing one method that knows everything, which means it's hard to test, hard to read, and risky to change.

The pattern in this guide is the antidote, and it's older than any framework you're using. Split the work into stages. Give each stage one job and a small interface. Let each stage depend on the idea of the next stage, not its guts. Then assemble the whole chain in exactly one place. That one place is the only part of the program that knows the concrete types; everything else just talks to interfaces.

The Stages

The Four Jobs and Their Boundaries

Name the jobs out loud and the layers fall out almost for free.

So what are the stages? Say the task slowly and they announce themselves. Read records from a file, that's one job: an Import Service that owns the boundary to the outside world. Store them and read them back, that's persistence, and it splits into two pieces, a Persistence Context that knows how to talk to the database engine and a Repository that gives the rest of the code a tiny, database-free way to ask for records. Run those steps in order and present the result, that's an Orchestration Service, the conductor that doesn't do any of the real work itself but knows the sequence.

And the record that flows through all of it? That's the Domain Model, a plain little class with no behavior, just fields. It's the common currency every layer trades in.

Notice that each stage is defined by a boundary, not by an algorithm. The Import Service is "where bytes from outside become records." The Repository is "where code stops caring how storage works." Drawing those boundaries well is the whole skill, and it's what makes the layers swappable later.

The Glue

Composition Root and Dependency Injection

If the layers never name each other, something has to introduce them. That something is one small block of startup code.

There's a catch in everything we just said. If the Orchestration Service only knows the IRepository interface, who decides which concrete repository it actually gets? The layers deliberately don't construct each other, so the wiring has to live somewhere outside all of them. That somewhere is the composition root: the single startup location where you register every interface against its real implementation and let a container build the object graph for you.

This is dependency injection, and the phrase sounds heavier than the idea. A class lists what it needs in its constructor and trusts that someone will hand those things over. It never reaches out and builds a database connection itself. The container does the handing-over. The payoff shows up the first time you write a test: pass a fake repository into the orchestrator's constructor and the real database never gets touched. Same wiring mechanism, different parts plugged in.

What you gain

Each layer is testable in isolation, any implementation can be swapped without editing its callers, and there's exactly one file to read when you want to know how the program is assembled.

What it costs

More files, more interfaces, and a layer of indirection. For a genuinely trivial script that's overhead. The pattern earns its keep the moment the script grows a second reason to change.

That tension, ceremony now versus flexibility later, is the central trade-off of the whole pattern, and we'll keep coming back to it. For the rest of the guide we build the layers one at a time, in dependency order, then meet at the composition root to snap them together. Next up, though, is the real codebase all of this was drawn from, so you can see the concrete version before we generalize it.
