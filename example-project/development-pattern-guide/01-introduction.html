Chapter 1

Introduction

What layered architecture with dependency injection is, the problem it solves, and the trade-offs it makes.

The Pattern

Layered Architecture with Dependency Injection

A single sentence to anchor everything that follows.

This guide teaches one pattern: split an application into layers that each own a single concern, define the boundary between layers as an interface, and let a single composition root construct the concrete implementations and inject them into the objects that depend on them. Nothing constructs its own dependencies; everything receives them.

The example the pattern is drawn from is small on purpose. A console program reads records from a JSON file, stores them in a database through an ORM, reads them back, and prints them. That is enough work to need four distinct responsibilities (model, persistence, import, orchestration) and a place to wire them together. Those five pieces are the whole pattern, and every chapter after this one builds one of them.

Motivation

The Problem It Solves

Why a single file of procedural code stops working as a program grows.

The naive version of this program is one function: open the file, parse the JSON, open a database connection, write the rows, read them back, print them. It works, and for a script that runs once it is the correct amount of code. It stops being correct the moment any part of it needs to change independently of the others.

Consider the pressures that arrive over time. The data might come from an HTTP endpoint instead of a file. The store might move from an in-memory database to a real one. You might want a unit test that checks the orchestration logic without touching a disk or a database at all. In the single-function version, every one of these changes edits the same code, and none of them can be tested in isolation, because the parsing, the persistence, and the control flow are fused together.

Layering separates these concerns so each can change on its own. Dependency injection is the mechanism that keeps them separated: because a layer receives its collaborators through its constructor rather than creating them, you can hand it a real collaborator in production and a fake one in a test, and the layer cannot tell the difference.

Structure

The Five Roles

The participants in the pattern and the responsibility each one holds.

The pattern assigns every line of code to one of five roles. The names below are the general-purpose names this guide uses; each later chapter builds one role.

Domain Model

A plain object describing the data that moves through the system. It carries no behavior, only fields and the attributes that map it to storage and to the external format.

Database Context

The ORM's view of the store. It declares which entities are persisted and configures how each one maps to a table.

Repository

A small interface that exposes only the persistence operations the application needs, hiding the ORM behind it so callers never see it.

Import Service

The boundary to the outside world. It reads the external source, deserializes it, and produces domain models.

Orchestration Service

The use case. It calls the import service and the repository in order and turns their results into the program's output.

Composition Root

The one place that knows the concrete types. It registers each interface against its implementation, builds the provider, and starts the run.

Trade-offs

What the Pattern Costs and What It Buys

An honest account of the indirection you take on and the freedom you gain.

This pattern is not free. Every seam becomes an interface, so a program that could be one file becomes a handful of small ones, and reading it top to bottom no longer shows the full control flow in one place. For a one-off script this overhead is not worth paying, and you should not pay it.

What you buy is substitution. Because each layer depends on the interface of the layer below it and not the concrete class, you can replace the implementation without touching the caller. The in-memory database becomes a real one by changing a single registration. The file importer becomes an HTTP importer by writing a new class that implements the same interface. A test replaces the repository with an in-memory list. The dependency arrows all point inward toward the interfaces, and the composition root is the only code that knows the concrete graph.

The rule of thumb: adopt this pattern when more than one of these substitutions is plausible, and skip it when none of them is.
